# Problem Query 01: Latest Anderson Lab Sequencing Sample

## Initial Query

> Give me the irods path, study id and sample supplier name of the most recently sequenced sample from Anderson lab.

## Approach I Took

I interpreted "Anderson lab" as studies whose `faculty_sponsor` contains `Anderson`, because the MLWH tools do not expose a first-class "lab" concept. I first resolved `Anderson` as a person term and found several possible matches, including `Carl Anderson`, `Carl Anderson/Jeff Barrett`, and Anderson-named study users. I then used the faculty-sponsor route and treated Anderson-sponsored studies as the relevant scope.

The main path was:

1. Resolve `Anderson` with `mlwh_resolve_person`.
2. List Anderson-sponsored studies with `mlwh_studies_for_faculty_sponsor`.
3. Page through all 91 matching studies.
4. Call `mlwh_study_overview` for those studies in batches and compare `newest_data_added`.
5. Identify study `7682` as the newest Anderson-sponsored study by study-level iRODS data timestamp:
   `2026-06-25T01:57:40Z`.
6. Try to use `mlwh_samples_with_data_for_study` with tight `since`/`until` windows around that timestamp to identify the sample.
7. When that returned no samples for study `7682`, inspect `mlwh_irods_paths_for_study`, `mlwh_study_manifest`, `mlwh_runs_for_study`, `mlwh_run_overview`, `mlwh_run_status`, and `mlwh_sample_progress`.
8. Return a caveated row from the latest Illumina run I could identify in study `7682`.

The final answer I gave was one tied latest Illumina sample row:

- iRODS path: `/seq/illumina/runs/51/51945/lane6/plex45/51945_6#45.cram`
- Study ID: `7682`
- Supplier name: `SOAD0950`
- Sample: `7682STDY15664546`

That answer was not as strong as it should have been. It was a best-effort answer from the available MCP tools, not a direct proof of the actual most recent study-scoped iRODS row.

## Problems With The MCP Server Offerings

The server had good building blocks, but no tool matched the query shape directly. The user asked for "the most recently sequenced sample from Anderson lab" and wanted fields from a joined row: iRODS path, study id, and sample supplier name. The MCP surface made me reconstruct that manually from person tools, study lists, study overviews, iRODS lists, manifests, run tools, and sample progress tools.

The biggest problem was that `mlwh_study_overview` exposes `newest_data_added`, but not the iRODS row, sample, supplier name, path, product id, or platform responsible for that maximum timestamp. It can tell me that study `7682` has a latest study-scoped iRODS timestamp of `2026-06-25T01:57:40Z`, but it cannot tell me which data object caused that max.

The natural follow-up, `mlwh_samples_with_data_for_study` with a `since`/`until` window around `2026-06-25T01:57:40Z`, returned no samples for study `7682`, even though the overview reported that timestamp. The same style of window query did work for study `6957`, returning 11 samples around `2026-06-24T15:32:21Z`. That suggests a denominator or join mismatch: `StudyOverview.newest_data_added` is based on study-scoped iRODS rows, while `samples-with-data` returns distinct linked samples. If the newest iRODS row is not linkable through the sample/library path used by `samples-with-data`, the tools cannot bridge from the timestamp back to the requested sample fields.

The iRODS path listing did not include the iRODS `created` timestamp. `mlwh_irods_paths_for_study` returns useful fields such as `id_product`, `collection`, `data_object`, `irods_path`, `id_sample_tmp`, `name`, `id_run`, and `platform`, but not `created`. The upstream `wa` `IRODSPath` type confirms that `created` is not part of the response. Without `created`, I could not sort or filter returned paths by recency.

The study iRODS list is ordered by product/sample identity, not by recency. The upstream query orders by `id_iseq_product, id_sample_tmp`, so paging to the end or beginning is not a recency operation.

The manifest endpoint was also not enough. `mlwh_study_manifest` is product-grained and useful for Illumina-style run/lane/tag rows, but it does not expose iRODS `created`, and it does not solve the cross-platform recency problem. In study `7682`, the iRODS list included non-Illumina or unmatched rows with `id_run: 0` and platform values such as `Ultimagen`. The study's `newest_data_added` was much later than the Illumina run overview timestamps I could inspect, so the true newest row may be outside the Illumina manifest/run path I used for the final workaround.

The run tools are Illumina-centric. `mlwh_runs_for_study`, `mlwh_run_overview`, and `mlwh_run_status` helped me identify Illumina runs `51930` and `51945`, but they could not resolve non-Illumina rows that may have caused the latest study-level iRODS timestamp.

The person/lab semantics were also ambiguous. `Anderson` resolved to a named faculty sponsor, a joint faculty sponsor, and several study-user records. "Anderson lab" might mean faculty sponsor `Carl Anderson`, studies where Carl Anderson is an owner, a data access group containing `ca3`, or a broader operational lab definition. The MCP server did not provide a first-class lab resolver or a route that states which interpretation should be used.

Finally, the available generic escape hatch, `mlwh_call_endpoint`, could only call existing `wa mlwh` endpoints. It could not help because the missing capability was not merely absent from the curated MCP wrapper; it was absent from the upstream endpoint catalogue too.

## Attempted Workarounds

I paged through `mlwh_studies_for_faculty_sponsor` rather than relying on the first page. There were 91 Anderson-sponsored studies. I then called `mlwh_study_overview` across those studies in batches and compared `newest_data_added`. This found study `7682` as the newest study-level candidate. Nearby contenders included study `6957`, with `newest_data_added` at `2026-06-24T15:32:21Z`, and study `6278`, with `newest_data_added` at `2026-04-09T00:23:48Z`.

I queried `mlwh_samples_with_data_for_study` for study `7682` using exact and widened windows around `2026-06-25T01:57:40Z`, including one second, one minute, ten minutes, and a full UTC day. These all returned zero samples. I verified that the same endpoint worked for study `6957` by querying around its latest timestamp and receiving 11 samples.

I inspected `mlwh_irods_paths_for_study` for study `7682`. This confirmed the study had many iRODS paths and included both Illumina rows and non-Illumina-looking rows where `id_run` was `0` and `platform` was `Ultimagen`. However, the rows did not include `created`, so I could not identify the row that matched the study overview's max timestamp.

I used `mlwh_study_manifest` with `with_irods=true` for study `7682`. This exposed supplier names and iRODS paths for product rows and showed the Illumina transition from run `51930` to run `51945`. It still did not include iRODS creation timestamps, so it did not prove which sample was newest.

I used `mlwh_runs_for_study`, `mlwh_run_overview`, and `mlwh_run_status` to compare runs `51930` and `51945`. Run `51945` was the later Illumina run by run lifecycle and run iRODS date range. I then inspected manifest rows around the start and end of run `51945`, and checked `mlwh_sample_progress` for representative edge rows. This established a tied latest Illumina-run sample set, but it did not establish the true latest study-scoped iRODS row.

I also checked the local repository and upstream `wa` module source. The upstream `IRODSPath` response type does not carry `created`, and the study iRODS list query orders by product/sample identity rather than by `created`. That confirmed the limitation was in the available API surface, not just my MCP usage.

## Remaining Doubts

I am not confident that the row I returned is the actual most recently sequenced sample from Anderson lab. I am confident that study `7682` was the newest Anderson-sponsored study by `mlwh_study_overview.newest_data_added`, but I could not identify the exact iRODS row behind that timestamp.

The phrase "most recently sequenced" is itself underspecified. It could mean latest run start, run completion, QC completion, archival completion, data delivery to iRODS, or latest iRODS object creation. I treated iRODS `created` / `newest_data_added` as the best proxy because the user asked for an iRODS path, but the MCP server did not let me carry that timestamp through to a path and supplier name.

There may be ties. A sequencing run usually contains many multiplexed samples with the same run-level timestamp. A correct endpoint should either return all rows tied at the max timestamp or document deterministic tie-breaking.

The lab scope may be wrong. I used `faculty_sponsor contains Anderson`; that includes `Carl Anderson/Jeff Barrett` and may exclude studies where Anderson is a study user but not the sponsor. It may also include validation or inactive studies depending on the user's mental model of "lab".

The actual newest row may be a non-Illumina or unmatched platform row in study `7682`, not one of the Illumina run `51945` manifest rows I used for the final answer.

## Improvement Ideas

The best fix would be a direct upstream query for this question shape. `wa mlwh` should expose an endpoint such as:

`GET /studies/faculty-sponsor/:name/latest-irods?limit=...`

or a more general:

`GET /irods/latest?faculty_sponsor=Anderson&limit=...`

The response should include `created`, `irods_path`, `id_study_lims`, study name, faculty sponsor, sample name, supplier name, `id_sample_tmp`, product id, platform, run id when available, lane/tag when available, and an explicit flag when the iRODS row cannot be joined to a linked sample. It should sort by `created DESC` and return all ties or document tie-breaking.

At minimum, the iRODS path endpoints should include the iRODS `created` timestamp and support `order_by=created_desc`, `since`, and `until`. Then the MCP server could answer this query by listing Anderson studies, fetching the newest path per candidate study, and doing a deterministic final comparison without guessing.

`StudyOverview` and `samples-with-data` should use clearly aligned recency semantics or expose both concepts separately. For example, `StudyOverview` could return `newest_study_irods_added` and `newest_linked_sample_data_added`, with counts for unlinked iRODS rows. That would prevent the current trap where the overview reports a max timestamp that cannot be followed through `samples-with-data`.

The server should add a "latest data for study" endpoint:

`GET /study/:id/irods/latest?limit=...`

This would return timestamped iRODS rows joined to sample and supplier fields. The current manifest endpoint is close for Illumina products, but it is product-grained, omits `created`, and does not cleanly cover cross-platform rows.

The MCP server should also offer a batch overview or batch latest endpoint. Answering the Anderson query required up to 91 per-study checks. A batch endpoint accepting study ids, or a sponsor-scoped aggregate endpoint, would make the query fast and avoid manual paging mistakes.

Lab/person routing should be more explicit. A `resolve_lab` or sponsor-scoped latest-data tool could state whether it uses faculty sponsor, study-user membership, data access group, or another lab definition. If "lab" cannot be resolved unambiguously, the tool should return candidate scopes and counts before the data query runs.

Finally, the workflow guidance should call out this exact pattern: when a user asks for "latest sample/path/supplier", do not use `study_overview` alone. Use a timestamped latest-iRODS endpoint that returns the row-level fields needed for the answer. With such an endpoint, this question would become one or two tool calls instead of a long manual investigation with unresolved uncertainty.
